# i18n & Accessibility

These two are non-negotiable. CI catches some violations
(`svelte-check`, ESLint a11y rules, `frontend-remove-unused-components`)
but most are on you. This page is the contract.

## i18n

### Single source of truth — `en.json` only

- Strings live in
  [`src/frontend/src/lib/i18n/en.json`](../../../src/frontend/src/lib/i18n/en.json).
- All other locales (`ar.json`, `cs.json`, `de.json`, `es.json`,
  `fr.json`, `hi.json`, `it.json`, `ja.json`, `ko-KR.json`, `pl.json`,
  `pt.json`, `ru.json`, `vi.json`, `zh-CN.json`) are **machine-translated
  from `en.json`** by the `auto-update-i18n` workflow. **Never hand-edit
  them** — your changes will be overwritten on the next run.
- The typed shape `I18n` and key helpers are generated by:

  ```bash
  npm run i18n   # = npm run i18n:types && npm run i18n:keys
  ```

  This regenerates
  [`src/frontend/src/lib/types/i18n.d.ts`](../../../src/frontend/src/lib/types/i18n.d.ts)
  and the keys file under `src/frontend/src/lib/i18n/`. Commit them
  together with `en.json`. Never edit the generated `.d.ts` by hand.

### Reading strings

In a component:

```svelte
<script lang="ts">
	import { i18n } from '$lib/stores/i18n.store';
</script>

<button>{$i18n.core.text.save}</button>
```

In a `.ts` module:

```ts
import { i18n } from '$lib/stores/i18n.store';
import { get } from 'svelte/store';

const msg = get(i18n).send.error.amount_too_low;
```

Always reference the typed key via property access — never construct keys
dynamically (`$i18n[someVar]`). Property access lets `svelte-check` and
the orphan-key tooling find dead keys.

### Placeholders

Existing keys use `$placeholder` syntax (e.g.
`"more_items": "+$items more"`). Substitute via `replacePlaceholders` from
`$lib/utils/i18n.utils`:

```ts
import { replacePlaceholders } from '$lib/utils/i18n.utils';

const label = replacePlaceholders($i18n.core.text.more_items, { $items: `${count}` });
```

### Naming keys

Use snake-case dotted paths grouped by feature, then concern, then
purpose. Look at the existing tree in `en.json` before adding. Common
top-level buckets: `core`, `auth`, `send`, `receive`, `swap`, `tokens`,
`networks`, `settings`, `wallet_connect`, `transactions`, `nfts`,
`rewards`, `ai_assistant`, `convert`, `agreements`, etc.

Within a bucket, common sub-buckets:

- `text.*` — labels, headings, body copy.
- `actions.*` — button / CTA copy.
- `error.*` — error messages.
- `placeholder.*` — input placeholders.
- `alt.*` — `alt` text and `aria-label`s.
- `info.*` — banners, tooltips, descriptive copy.

Workflow for adding a new key: see
[`workflows/add-i18n-key.md`](./workflows/add-i18n-key.md).

### Forbidden

- Hard-coded English in any user-visible context (text, `aria-label`,
  `title`, `alt`, `placeholder`, button copy, error messages).
- Concatenating sentences with `+` — pluralisation and grammar break in
  other locales. Use a parameterised key with `$placeholder`s.
- Editing any non-`en` locale JSON file directly (overwritten by CI).
- Editing `i18n.d.ts` or the generated keys file by hand.

### Allowed (no i18n needed)

- Test IDs (`data-tid`).
- Internal log messages (`consoleError` / `consoleWarn` payloads).
- Brand names (e.g. `OISY`, `Internet Computer`, `Bitcoin`, `Ethereum`,
  `WalletConnect`).
- Token symbols (`BTC`, `ETH`, `ICP`, `SOL`, …).

## Accessibility (a11y)

Baseline rules; ESLint a11y rules and `svelte-check` catch some violations.

### Semantics

- **Use the right element.** `<button>` for actions, `<a>` for navigation,
  `<label>` for inputs, `<h1..h6>` for hierarchy in order, `<nav>` /
  `<main>` / `<aside>` / `<header>` / `<footer>` for landmarks.
- **No bare clickable `<div>`.** If you absolutely need a `<div>` /
  `<span>` for visual reasons, set `role`, `tabindex="0"`, and keyboard
  handlers (`onkeydown` for Enter / Space). Prefer the right element.

### Labels

- Every form input has a programmatic label. Either:
  - A `<label for="…">` paired with `id`, or
  - `aria-label` / `aria-labelledby`.
- Icon-only buttons need `aria-label` (or visible text via an `sr-only`
  span). Pull the label from `$i18n.<bucket>.alt.*`.
- Decorative icons → `aria-hidden="true"`.

### Focus & keyboard

- Don't kill focus outlines.
- Modals / drawers / popovers — use the existing `@dfinity/gix-components`
  primitives (or `$lib/components/ui/`). They handle focus trap, escape,
  click-outside. Do not roll your own.
- Modal close button must be reachable by keyboard.
- For lists where the user picks one of N items, focus order matches
  visual order.

### Color & contrast

- Never communicate state with color alone — pair it with an icon, label,
  or shape.
- Use the project's design tokens / utility classes. Search the closest
  neighbour and follow it; don't hard-code hex.

### Images / media

- Meaningful images: `alt="…"` describing the content (typically from
  `$i18n.<bucket>.alt.*`).
- Decorative images: `alt=""` (empty, not omitted).
- SVG icons: see [Icons](#icons) below.

### Routing

- Page-level navigation should update the document title (handled by
  layout / `+page.ts` — don't bypass).
- `target="_blank"` requires `rel="noopener noreferrer"`.

### Icons

The codebase uses several icon sources (see
[stack-and-patterns.md](./stack-and-patterns.md#icons)). Whatever the
source: decorative icons inside a labelled control get `aria-hidden="true"`,
and icon-only controls get `aria-label="…"` on the button (not on the
icon).

### ARIA — minimum surface

Use ARIA only when HTML semantics are insufficient. The `gix-components`
primitives already set the right roles / `aria-*` attributes for menus,
dialogs, tooltips, etc. — don't override them.

### Testing accessibility

- For non-trivial interactive components, write a Vitest spec that:
  - Renders the component.
  - Asserts the accessible name is present (e.g.
    `screen.getByRole('button', { name: /submit/i })`).
  - Drives the interaction with keyboard events when keyboard support
    matters.
- See [`testing.md`](./testing.md).

## Quick checklist (every UI change)

- [ ] All visible strings come from `$i18n.*`.
- [ ] No `text-#hex`, no inline `style="color:…"` — use the existing token
      classes.
- [ ] Every interactive element is a real `<button>` / `<a>` / primitive.
- [ ] Every input has a label or `aria-label`.
- [ ] Icon-only buttons have an accessible name.
- [ ] Decorative icons / images have `aria-hidden="true"` / empty `alt=""`.
- [ ] State is conveyed by more than colour.
- [ ] External links: `rel="noopener noreferrer"` on `target="_blank"`.
- [ ] If untrusted HTML is rendered: sanitize first, then `{@html …}`.
- [ ] `npm run i18n` was run after editing `en.json`, and the generated
      files are committed.
